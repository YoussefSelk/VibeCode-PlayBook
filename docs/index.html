Why It Works

Small docs, clear routing, real verification

Token-efficient by default

The playbook teaches agents to load only the context needed for the current pass instead of dumping the whole repo into every task.

Good for humans too

The same structure that reduces token waste also makes onboarding calmer and easier for new contributors.

Built for real repos

Install it into an existing project and keep your current stack. The playbook should adapt to the repo, not the other way around.

Assistant-agnostic workflow

The role model works with Codex, GitHub Copilot, Claude, and other coding assistants. If a tool has no sub-agents, use the same roles as explicit phases in one chat.

One installer, smaller setup

The installer now asks which assistant you use, then downloads only the matching pack instead of every tool-specific file.

Shared agent layer

A neutral .agents/ layer is installed for shared role definitions, while tool-specific files stay in their own pack.

Install

Use one installer and choose your assistant pack

Pick your assistant, then copy the command for your shell. If you omit the assistant flag, the installer shows a menu.

PowerShell installer for Codex

powershell 
iex "& { $(irm https://raw.githubusercontent.com/YoussefSelk/VibeCode-PlayBook/main/scripts/install.ps1) } -Assistant codex"

Bash installer for Codex

bash 
curl -fsSL https://raw.githubusercontent.com/YoussefSelk/VibeCode-PlayBook/main/scripts/install.sh | bash -s -- --assistant codex

Deploy

Publish the site through GitHub Pages

The repo ships with a GitHub Actions workflow that deploys the contents of docs/ to Pages on pushes to main.

What is wired up

  • .github/workflows/deploy-pages.yml uploads docs/ as the Pages artifact
  • pushes to main rerun the publish job when docs or the workflow change
  • no build step is required because the site is plain HTML, CSS, and JavaScript

What you still need in GitHub

  • enable GitHub Pages for the repository if it is not already on
  • set the source to GitHub Actions in the repository Pages settings
  • confirm the workflow has permission to deploy Pages artifacts

After the first successful run, review the live URL and confirm the copy, section links, and machine-readable files match the repo.

FAQ

Common onboarding questions

Is this a full app framework?

No. It is a workflow layer you install into an existing repo so your agents and contributors follow a clearer operating model.

Why does it have both human docs and LLM docs?

Because this repo is meant to coordinate both. Humans need a clear mental model, and LLMs benefit from compact, explicit entry points.
